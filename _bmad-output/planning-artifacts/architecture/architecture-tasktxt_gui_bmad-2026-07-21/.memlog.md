- (constraint) Brownfield native macOS 13+ SwiftUI app; TasksTxtCore owns todo.txt parsing and FileSystemTaskDocumentStore owns persistence.
- (constraint) No plugin development is authorized until the user approves the completed evaluation report.
- (decision) Use a two-layer plugin model: command plugins produce validated TaskCommands; controlled-page plugins produce a versioned declarative UI document rendered only by host-owned SwiftUI components. This prevents native view ABI coupling and arbitrary host UI mutation.
- (decision) Both layers share one manifest, capability gate, runner, data snapshot, command protocol, lifecycle, and compatibility policy; controlled pages add no direct data or mutation path.
- (decision) Third-party plugins never receive TaskStore, TaskDocumentStore, file URLs, NSView, SwiftUI View, shell, environment, or unrestricted network access.
- (decision) All mutations are host-owned, revision-checked TaskCommands; page actions emit command intents through the same gate as command plugins.
- (decision) Controlled pages use a finite allowlist of host components and actions; unknown schema nodes, properties, events, and capabilities fail closed.
- (decision) Public third-party execution requires process isolation plus explicit sandbox entitlements and resource limits; XPC alone is only a process boundary.
- (assumption) [ASSUMPTION] Version 1 pages may be added as separate navigation destinations but cannot replace, wrap, reorder, or inject controls into built-in List, Quadrants, Scratchpad, Statistics, Settings, onboarding, or global capture views.
- (assumption) [ASSUMPTION] Version 1 component allowlist is Page, Section, Text, TaskList, StatCard, BarChart, Button, Form, TextField, Picker, Toggle, Divider, Spacer, and EmptyState.
- (assumption) [ASSUMPTION] Version 1 has no arbitrary network capability and no remote JavaScript or remote UI schema; installed packages are local and immutable while enabled.
- (question) Should controlled plugin pages appear as top navigation tabs, a Plugins hub, or command-palette destinations? Recommendation: Plugins hub plus optional pinned top-level destinations to avoid crowding the current five tabs.
- (question) Is the first release trusted local plugins only, or public third-party installation? This changes whether XPC sandbox, signing, revocation, and update security are release blockers.
- (question) Should a controlled page be allowed to request all-task snapshots, or only host queries with bounded fields and result counts? Recommendation: bounded host queries by default; all-task access remains a separately confirmed capability.
- (version) Verified local toolchain: Xcode 26.5 build 17F42, Apple Swift 6.3.2 compiler in Swift language mode 5, project deployment target macOS 13.0.
- (version) Verified Apple JavaScriptCore documentation currently exposes JSContext/JSVirtualMachine and native-object bridging; use system framework rather than bundling a runtime.
- (version) Verified Apple XPC documentation describes launchd-managed services and privilege isolation, but sandbox/entitlement policy remains an explicit architecture responsibility.
- (version) Verified Apple SwiftUI documentation supports declarative view hierarchies, host-owned layout containers, NavigationStack, and value-based destinations; plugin UI remains a host renderer, not executable SwiftUI supplied by plugins.
- (constraint) Existing TaskWorkspace already defines TaskHandle and a limited TaskCommand enum, providing a brownfield seam to expand rather than replacing TasksTxtCore.
- (constraint) Existing TaskDocumentStore generation is process-local and archiveCompleted spans two files without a crash-safe transaction; plugin launch is gated on revision validation and journaling work described in the report.
- (assumption) [ASSUMPTION] Controlled-page schema major version 1 supports at most 200 nodes per page, nesting depth 8, 100 task rows per query page, and a 256 KB rendered document payload.
- (assumption) [ASSUMPTION] Page refresh is pull-based on entry/user action plus debounced host invalidation; plugin code does not stay resident merely because its page is visible.
- (decision) Use a Plugins hub with optional pinning as the navigation seed; plugin destinations are lightweight IDs owned by the host, not task models or plugin-supplied view objects.
- (decision) Use one schema major version and additive minor features; unknown major rejects the page, unknown nodes or actions fail the page closed, and deprecated nodes remain supported for at least one app minor release after warning.
- (correction) Aligned the page schema example with the architecture rule by adding exactly one explicit `page` root; added the current report to spine sources and corrected one spacing issue.
- (review) Rubric, current-technology, adversarial-consistency, structural editorial, and prose editorial reviews completed. No blocking finding remains; public-plugin implementation remains gated on the XPC/Sandbox spike.
- (decision) Architecture spine finalized as a reviewed decision document. Product approval is still pending, so no plugin implementation has started.
